Раздел 16.1. Pinia: архитектура, плагины, persist

Разбираем, как строить Pinia для реального проекта: домены, плагины и надежное сохранение состояния.

Цель урока

Освоить ключевые идеи раздела и закрепить их через короткую пошаговую практику.

Теория простыми словами

1) Архитектура store по доменам

Store лучше делить по зонам ответственности: auth, cart, catalog, а не хранить все в одном месте.

2) Плагины Pinia

Плагин подключает общую функциональность ко всем store: логирование, аудит, метки времени, persist-поведение.

3) Persist состояния

Сохраняем только нужные поля и безопасно восстанавливаем их при перезагрузке страницы.

4) Не дублируй state

Если одно и то же состояние хранится в разных store, быстро появляется рассинхрон и сложные баги.

Практика по шагам

Цель: вынести состояние магазина по доменам, добавить плагин и сохранение корзины.

Шаг 1. Раздели store по доменам

// stores/auth.js
export const useAuthStore = defineStore('auth', { /* ... */ })

// stores/cart.js
export const useCartStore = defineStore('cart', { /* ... */ })

Зачем шаг: легче поддерживать и тестировать состояние.

Шаг 2. Подключи pinia-plugin с меткой обновления

function timestampPlugin({ store }) {
  store.$subscribe(() => {
    store.lastUpdatedAt = Date.now()
  })
}

const pinia = createPinia()
pinia.use(timestampPlugin)

Что изменится: можно отслеживать, когда состояние обновлялось в последний раз.

Шаг 3. Сохраняй только нужные поля корзины

cartStore.$subscribe((_mutation, state) => {
  localStorage.setItem(
    'cart',
    JSON.stringify({ items: state.items, coupon: state.coupon })
  )
})

Зачем шаг: в storage не должно попадать лишнее и чувствительное.

Шаг 4. Восстанови persist при старте

const saved = localStorage.getItem('cart')
if (saved) {
  const parsed = JSON.parse(saved)
  cartStore.$patch({
    items: Array.isArray(parsed.items) ? parsed.items : [],
    coupon: parsed.coupon ?? null
  })
}

Что изменится: корзина не теряется после обновления страницы.

Шаг 5. Убери дублирование состояния

// Плохо: user хранится в authStore и profileStore одновременно
// Хорошо: единый источник в authStore, profileStore берет данные через getter
const authStore = useAuthStore()
const userName = computed(() => authStore.user?.name ?? '')

Зачем шаг: один источник правды снижает риск рассинхрона.

Практическое задание

Собери архитектуру состояния мини-магазина на Pinia.

  1. Создай доменные store: auth, cart, ui.
  2. Подключи плагин, который фиксирует lastUpdatedAt.
  3. Добавь persist для корзины и безопасное восстановление.
  4. Проверь, что одно и то же состояние не дублируется в двух store.
  5. Подготовь короткий список "что хранить в store, а что локально в компоненте".

Типичные ошибки

  • Один огромный store вместо доменной структуры.
  • Persist всех полей подряд, включая лишние данные.
  • Дублирование одного состояния в разных store.

Типичные ошибки

  • Складывать в persist весь store вместо точечного сохранения нужных полей.
  • Не валидировать данные из localStorage перед восстановлением состояния.
  • Размывать границы доменов и смешивать auth/cart/ui-логику в одном store.
