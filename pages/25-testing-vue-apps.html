Раздел 22. Тестирование Vue-приложений

Учимся уверенно проверять качество: unit, component и e2e тесты на практических сценариях.

Цель урока

Научиться проверять важные сценарии приложения и заранее ловить ошибки до выпуска.

Теория простыми словами

1) Что тестировать в первую очередь

Сначала критичные пользовательские сценарии: логин, форма, корзина, сохранение данных и ключевая бизнес-логика.

2) Unit-тесты composables/stores

Проверяют чистую логику без браузера: функции, вычисления, действия store, валидацию.

3) Component tests (Vue Test Utils)

Проверяют поведение компонента: рендер, ввод, клики, emit-события, условные блоки.

4) E2E (Playwright/Cypress)

Проверяют пользовательский путь целиком: от открытия страницы до успешного результата.

5) Моки API и стратегия тестов

Моки делают тесты стабильнее. Команда должна договориться, что тестируем на уровне unit, component и e2e.

Практика по шагам

Практическое задание: покрыть тестами мини-приложение задач.

Мостик: начинаем с быстрых unit-проверок, затем поднимаемся к компонентным и завершаем целостным сценарием поведения.

Шаг 1. Определи приоритетные сценарии

Критичные сценарии:
1) Добавление задачи
2) Переключение статуса done
3) Удаление задачи
4) Ошибка загрузки с API

Шаг 2. Unit-тест composable

import { describe, it, expect } from 'vitest'
import { useCounter } from './useCounter'

describe('useCounter', () => {
  it('увеличивает значение', () => {
    const { count, inc } = useCounter()
    inc()
    expect(count.value).toBe(1)
  })
})

Шаг 3. Unit-тест Pinia store

import { setActivePinia, createPinia } from 'pinia'
import { useTodoStore } from './todoStore'

beforeEach(() => setActivePinia(createPinia()))

it('добавляет задачу', () => {
  const store = useTodoStore()
  store.addTask('Новая задача')
  expect(store.items.length).toBe(1)
})

Шаг 4. Component test (Vue Test Utils)

import { mount } from '@vue/test-utils'
import TodoForm from './TodoForm.vue'

it('отправляет событие submit с текстом', async () => {
  const wrapper = mount(TodoForm)
  await wrapper.get('input').setValue('Купить молоко')
  await wrapper.get('form').trigger('submit.prevent')

  expect(wrapper.emitted('submit')?.[0]).toEqual(['Купить молоко'])
})

Шаг 5. E2E-сценарий (концептуально)

test('пользователь добавляет задачу', async ({ page }) => {
  await page.goto('/todos')
  await page.fill('[data-test=todo-input]', 'Задача E2E')
  await page.click('[data-test=todo-submit]')
  await expect(page.locator('text=Задача E2E')).toBeVisible()
})

Шаг 6. Моки API

vi.spyOn(global, 'fetch').mockResolvedValue({
  ok: true,
  json: async () => ([{ id: 1, title: 'Mock task' }])
} as Response)

Шаг 7. Мини-стратегия тестов в команде

- Unit: бизнес-логика и вычисления
- Component: поведение UI-компонентов
- E2E: ключевые пользовательские пути
- Не дублировать одинаковые проверки на всех уровнях

Практическое задание

Покрой мини-приложение задач тестами.

  1. Напиши unit-тест на composable или store.
  2. Напиши component test формы добавления задачи.
  3. Добавь e2e сценарий "добавить задачу".
  4. Смоделируй ошибку API и проверь fallback UI.
  5. Собери короткий test-plan для команды.

Мини-практика: тест на отправку формы и проверка emit-события.

Типичные ошибки

  • Писать тесты, привязанные к внутренней реализации, а не к поведению.
  • Пытаться покрыть все E2E сразу, игнорируя более быстрые unit/component тесты.
  • Не мокать внешние API и получать нестабильные результаты тестов.

Проверка знаний

Короткий тест по ключевым идеям этого урока.
