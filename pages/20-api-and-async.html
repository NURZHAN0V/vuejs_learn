Раздел 17. Работа с API и асинхронностью

Учимся получать и отправлять данные на сервер: обработка статусов, ошибок, отмены запросов и оптимистичных обновлений.

Цель урока

Научиться строить устойчивый поток работы с API: корректно показывать состояния запроса, обрабатывать ошибки, отменять лишние запросы и безопасно обновлять UI.

Теория простыми словами

1) fetch/axios в Vue-приложении

Любой HTTP-клиент подходит. Важно не сам инструмент, а правильная обработка состояний и ошибок.

2) Состояния запроса: idle/loading/success/error

Данные и статус нужно хранить отдельно, чтобы UI показывал корректный экран в каждый момент.

3) Retry и отмена через AbortController

Повтор полезен при временных сбоях. Отмена нужна, чтобы не было гонок запросов и лишней нагрузки.

4) Оптимистичные обновления

UI сразу показывает результат, а затем подтверждает с сервером. При ошибке откатываем состояние.

5) Пагинация, фильтрация, debounce

Для больших списков запросы нужно ограничивать и не отправлять на каждый символ без задержки.

Практика по шагам

Практическое задание: CRUD-список задач с сервером.

Мостик: в каждом шаге добавляется один аспект запроса — загрузка, ошибка, отмена и повтор, чтобы итоговый поток был устойчивым.

Шаг 1. Базовое состояние данных и статуса

import { ref } from 'vue'

const tasks = ref([])
const status = ref('idle') // idle | loading | success | error
const errorMessage = ref('')

Шаг 2. Загрузка списка через fetch

const loadTasks = async () => {
  status.value = 'loading'
  errorMessage.value = ''
  try {
    const res = await fetch('/api/tasks')
    if (!res.ok) throw new Error('Не удалось получить задачи')
    tasks.value = await res.json()
    status.value = 'success'
  } catch (e) {
    status.value = 'error'
    errorMessage.value = e.message
  }
}

Шаг 3. Retry при ошибке

const retryLoad = async (max = 2) => {
  for (let i = 0; i <= max; i += 1) {
    await loadTasks()
    if (status.value === 'success') break
  }
}

Шаг 4. Отмена запроса (AbortController)

let controller = null
const searchTasks = async (q) => {
  if (controller) controller.abort()
  controller = new AbortController()
  try {
    const res = await fetch(`/api/tasks?q=${encodeURIComponent(q)}`, { signal: controller.signal })
    tasks.value = await res.json()
  } catch (e) {
    if (e.name !== 'AbortError') errorMessage.value = e.message
  }
}

Шаг 5. Оптимистичное создание задачи

const createTask = async (title) => {
  const temp = { id: Date.now(), title, done: false, _temp: true }
  tasks.value.unshift(temp)
  try {
    const res = await fetch('/api/tasks', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ title })
    })
    const saved = await res.json()
    tasks.value = tasks.value.map(t => (t.id === temp.id ? saved : t))
  } catch (e) {
    tasks.value = tasks.value.filter(t => t.id !== temp.id)
    errorMessage.value = 'Не удалось создать задачу'
  }
}

Шаг 6. Пагинация и debounce фильтра

import { ref, watch } from 'vue'
const page = ref(1)
const query = ref('')
let timer = null

watch(query, (value) => {
  clearTimeout(timer)
  timer = setTimeout(() => {
    searchTasks(value)
  }, 400)
})

Зачем шаг: не отправлять запрос на каждый символ и не перегружать API. Что изменится: поиск станет стабильнее, а риск гонок при быстром вводе снизится.

Практическое задание

Собери CRUD-список задач с сервером.

  1. Реализуй чтение, создание, обновление и удаление задач.
  2. Раздели данные и статус запроса (loading/error/success).
  3. Добавь retry для загрузки.
  4. Добавь отмену запросов при быстром поиске.
  5. Реализуй оптимистичное добавление задачи.

Мини-практика: поиск по API с debounce 300–500мс.

Типичные ошибки

  • Смешивать данные и статус загрузки в одной переменной.
  • Не отменять старые запросы и получать гонки результатов.
  • Не обрабатывать сценарий ошибки и оставлять пользователя без понятного состояния UI.
